## 1. 项目概述

### 1.1 目标

构建一套**个人使用**的 A 股量化交易系统，具备以下核心能力：

1. **因子量化策略**：基于技术指标/基本面因子的传统多因子选股与信号生成
2. **LLM 事件驱动策略**：基于新闻/公告的大模型情感分析与事件捕捉
3. **实时监控**：盘中实时看板 + 新闻流监控
4. **信号推送**：以「¥XX.XX 买入/卖出 XXXXXX（股票名）」的格式推送到飞书和邮件
5. **回测验证** *(Phase 2)*：历史数据回测，评估策略表现（需累积足够历史数据后再开启）

### 1.2 不做什么 (Non-Goals)

| 明确排除 | 原因 |
|---------|------|
| 券商接口对接/自动下单 | 当前阶段不需要，后续可扩展 |
| 高频交易 (Tick级) | 中低频定位，不需要毫秒级延迟 |
| 港股/美股 **交易** | MVP 专注 A 股交易信号。全球市场数据（美股指数、港股、大宗商品、国际新闻）从 Day 1 即纳入系统，作为 A 股策略的信号输入源 |
| 自研交易引擎 | 复用开源回测框架 (Phase 2) |

### 1.3 用户画像

- 单人使用，运行在一台 7×24 在线的 VPS/服务器上
- 有编程能力（Python），可维护和迭代系统
- A 股散户投资者，中低频操作（持仓周期：天~周级别）

## 2. 核心约束与设计原则

### 2.1 约束条件

| 约束 | 说明 |
|------|------|
| **无券商接口** | 仅做信号推送，不做自动交易 |
| **成本敏感** | 数据源和 LLM 优先免费/低成本方案，效果验证后再升级 |
| **单机部署** | VPS 上 Docker Compose 部署，不要求分布式 |
| **A股交易规则** | T+1、涨跌停板（主板±10%/创业板±20%）、印花税+佣金 |

### 2.2 设计原则

| # | 原则 | 说明 |
|---|------|------|
| P1 | **复用优先** | 核心模块复用热门开源项目（pandas-ta, LiteLLM 等），不重复造轮子 |
| P2 | **Top-Down 开发** | 先定义模块接口 → 用 Fake/Mock 数据跑通全流程 → 再逐模块替换为真实实现 |
| P3 | **接口隔离** | 每个模块通过明确的 Python Protocol/ABC 定义输入输出，可独立替换 |
| P4 | **渐进增强** | 数据源/LLM/策略 均支持从免费版平滑切换到付费版，不改核心逻辑 |
| P5 | **可观测性** | 所有关键步骤有日志和指标，出问题能快速定位 |
| P6 | **字段降级** | 每个数据模型定义必填字段 vs 可选字段；可选字段缺失时使用默认值或跳过该维度计算，不阻塞整体流程 |
| P7 | **告警聚合与分级** | 相同接口、相同错误类型在 10 分钟内合并为一条告警；仅影响交易信号的错误为高优先级 (立即推送)，数据延迟/非关键接口为低优先级 (汇总到日报) |
| P8 | **敏感信息隔离** | API Key、Webhook URL、SMTP 密码等敏感信息通过环境变量 / Docker Secrets / `.env` (gitignored) 注入，配置文件中仅保留占位符 |

### 3.4 回测流 【Phase 2】

> 回测需要累积足够的历史数据（至少 120 个交易日）才有统计意义，且 MVP 初期策略未定型，因此**回测与评估整体移入 Phase 2**。MVP 阶段通过「信号事后验证」（对比信号发出后的实际涨跌）来评估信号质量。
>
> Phase 2 触发条件：系统稳定运行 + 日线数据累积 ≥ 120 个交易日。届时引入 `backtesting.py` + `quantstats`，流程为：SQLite 缓存日线 → 因子计算 → 策略信号 → 回测引擎 → 绩效报告。

## 4. 模块详细设计

### 4.1 数据基建层 (Layer 1)

#### 4.1.1 行情数据 Provider

| 属性 | 说明 |
|------|------|
| **职责** | 获取 A 股 K线/实时行情 + 国际市场行情（美股指数、港股指数、大宗商品），统一输出标准化行情数据 |
| **接口** | `MarketDataProvider` (ABC) |
| **数据频率** | A 股: 日线 (核心) + 实时快照 (盘中)；国际: 快照 |
| **缓存策略** | 日线: ✅ SQLite 缓存 120 交易日；实时快照: ❌ 不缓存，用完即弃；国际快照: ✅ SQLite 保留全部历史 |
| **关联数据流** | 盘中实时流 (实时快照 + 缓存日线拼接「今日伪日线」用于因子计算) + 盘后更新流 (增量同步当日日线) |
| **调用频率** | 国际快照: 盘前 08:00 一次；A 股日线增量: 盘后 15:30 一次 |

**AkShare 请求频率限制与容错策略：**

AkShare 底层基于爬虫抓取东财/新浪等网站数据，无官方 rate limit 声明，但实测高频调用会触发源站反爬（HTTP 403/连接超时）。需在 Provider 层统一实施以下策略：

| 策略项 | 配置 | 说明 |
|--------|------|------|
| **请求间隔** | 单接口调用间隔 ≥ 0.5s (可配置) | `time.sleep()` 或令牌桶限速，防止短时间大量请求触发反爬 |
| **批量拉取间隔** | 批量循环 (如遍历 300 只股票) 每股间隔 0.3-1s | 300 只 × 0.5s ≈ 2.5 分钟，可接受 |
| **重试策略** | 指数退避 (Exponential Backoff)，最多 3 次 | 间隔 2s → 4s → 8s，使用 `tenacity` 库 |
| **重试触发条件** | HTTP 403/429/5xx、`ConnectionError`、`ReadTimeout`、AkShare 返回空 DataFrame | 区分可重试错误 vs 不可重试错误 (如参数错误) |
| **超时设置** | 单次请求超时 30s | 防止慢请求阻塞整个数据拉取流程 |
| **熔断保护** | 连续失败 5 次后暂停该接口 5 分钟 | 避免被源站 IP 封禁，同时触发 BaoStock 降级 |

**BaoStock 降级 (Fallback) 机制：**

| 场景 | 降级行为 |
|------|----------|
| AkShare 日K线接口连续失败 | 自动切换 BaoStock `bs.query_history_k_data_plus()` 获取日K线 |
| AkShare 实时行情接口不可用 | BaoStock **不支持实时行情** → 跳过本轮实时更新，使用上一次缓存数据，记录告警 |
| AkShare 国际行情接口失败 | BaoStock 不覆盖国际数据 → 跳过本轮国际行情，使用上一次缓存快照 |
| AkShare 恢复 | 下一轮自动恢复为 AkShare（健康检查：每 5 分钟尝试一次 AkShare ping 请求） |

> **实现要点**：Provider 内部持有 `primary: AkShareClient` 和 `fallback: BaoStockClient`，通过 `@retry_with_fallback` 装饰器统一处理。降级事件写入日志 + 推送飞书告警。

#### 4.1.5 数据存储

| 属性 | 说明 |
|------|------|
| **选型** | **SQLite** (MVP) / DuckDB (性能升级时) |
| **数据量** | 自选股 ≤50 只 (推荐 15-30) × 日线 (初始 120 日 + 每日累积, 不删除历史); 沪深300 因子池约 300 只 × 同规模; 首年合计约 10 万行, SQLite 完全胜任 |
| **表结构** | daily_bars, fundamentals, news_items, signals, data_sync_log (backtest_results → Phase 2) |

> **选择 SQLite 而非 PostgreSQL 的理由**：单机部署、数据量可控、零运维。DuckDB 留作列式分析性能升级路径。

**增量数据更新策略：**

系统采用「首次全量 + 每日增量」模式，避免重复拉取历史数据，减少 AkShare API 调用量并加速每日更新：

| 数据类型 | 首次初始化 | 每日增量更新 | 数据窗口 |
|---------|-----------|-------------|----------|
| A 股日K线 | 拉取近 120 个交易日 (约半年) 全量数据 | 每日盘后只拉取当日新增 1 条 | SQLite 保留全部历史 (不删除); 因子计算时内存加载最近 120 交易日 (MA60 × 2) |
| 沪深300成分日K线 | 同上，300 只 × 120 日 ≈ 3.6 万行 | 300 只 × 1 日 = 300 行/天 | 同上 (保留全部历史, 内存加载 120 日) |
| 基本面数据 | 拉取最近 4 期财报 | 每日检查是否有新财报发布，有则增量更新 | 保留最近 8 期 |
| 新闻/公告 | 拉取近 7 天 | 每小时增量拉取新条目 (按时间戳去重) | 滚动保留 30 天 |
| 国际行情 | 首次全量当日快照 | 盘前 08:00 获取快照，存入 SQLite | 保留全部历史 |

**增量更新流程：**

```
每日盘后更新 (15:30 触发):
───────────────────────────────────────
1. 查询 SQLite: SELECT MAX(trade_date) FROM daily_bars WHERE symbol = ?
2. 若 max_date < today:
   a. AkShare 拉取 (max_date+1, today) 的增量数据 (通常 1 天)
   b. 失败 → 重试 3 次 → 降级 BaoStock
   c. INSERT 新数据到 SQLite
3. 更新元数据表: data_sync_log (symbol, last_sync, status, source)
───────────────────────────────────────
首次初始化 (或数据缺失时):
  → 检测到 max_date 为空或间隔 > 5 天
  → 触发全量拉取 120 交易日
  → 300 只 × 0.5s 间隔 ≈ 2.5 分钟完成
```

**元数据追踪表 (`data_sync_log`)：**

```sql
CREATE TABLE data_sync_log (
    symbol      TEXT NOT NULL,
    data_type   TEXT NOT NULL,  -- 'daily_bars' / 'fundamental' / 'news'
    last_sync   TEXT NOT NULL,  -- ISO 日期时间
    last_date   TEXT,           -- 最新数据日期
    source      TEXT NOT NULL,  -- 'akshare' / 'baostock'
    status      TEXT NOT NULL,  -- 'ok' / 'failed' / 'fallback'
    error_msg   TEXT,
    PRIMARY KEY (symbol, data_type)
);
```

> **缓存一致性**：每次增量更新后，对比 AkShare 返回的前一日收盘价与本地缓存是否一致（允许 ±1% 相对误差），若不一致说明存在复权调整或数据修正，仅重新拉取最近 30 个交易日数据（而非全量重新拉取，避免因复权调整触发大量请求）。建议表设计保留未复权价 + `adj_factor` 列，前端/因子引擎按需复权。

### 4.2 因子工程层 (Layer 2)

> **设计理念**：参考 QLib Alpha158 因子库的设计思路，将因子计算从“手动挑选几个指标”升级为“系统化因子库”。后续 Phase 3 可引入 RD-Agent 自动发现新因子。

#### 4.2.1 技术因子计算

| 属性 | 说明 |
|------|------|
| **职责** | 计算技术指标/因子 |
| **复用** | **pandas-ta** (Phase 1) + **QLib Alpha158 因子表达式移植** (Phase 2) |
| **输出** | 因子 DataFrame (行=日期, 列=因子值) |

**分阶段因子集：**

| 阶段 | 因子数量 | 来源 | 说明 |
|------|---------|------|------|
| **Phase 1 (MVP)** | ~20 个 | pandas-ta + 自研 | 基础技术指标，快速验证流程 |
| **Phase 2** | ~80 个 | QLib Alpha158 移植 | 将 Alpha158 因子表达式翻译为 pandas 实现，无需 QLib 运行时依赖 |
| **Phase 3** | 100+ | RD-Agent 自动生成 | 用 RD-Agent 自动发现新因子，每次运行生成 5-10 个僠选因子 |

**Phase 1 MVP 因子集 (~20个)：**

| 因子类别 | 具体因子 | 来源 |
|---------|---------|------|
| 趋势 | MA5/MA10/MA20/MA60, EMA12/EMA26 | pandas-ta |
| 动量 | RSI(14), MACD(12,26,9), ROC(10) | pandas-ta |
| 波动率 | ATR(14), 布林带(20,2), 历史波动率(20) | pandas-ta |
| 成交量 | OBV, VWAP, 量比 | pandas-ta |
| 自定义 | 乖离率(BIAS), 均线多头排列评分 | 自研 |

> **盘中「伪日线 Bar」构造规则**：盘中实时扫描时，当日K线尚未收盘，因子引擎无法直接使用。系统采用「119 天历史缓存 + 实时快照拼接」方式构造伪日线 Bar：
> - 构造公式：`O = 今日开盘价`, `H = max(缓存最高, 实时最高)`, `L = min(缓存最低, 实时最低)`, `C = 当前价`, `V = 当前成交量`
> - 盘中因子值为**近似监控值**，非精确信号。收盘后以真实日 K 线修正，重新计算因子。
> - **盘中可靠因子**：均线类 (MA/EMA)、乖离率 (BIAS)、RSI — 受成交量不完整影响小
> - **仅收盘可靠因子**：量能类 (OBV/VWAP)、布林带 — 需完整成交量/收盘价才有意义

**Phase 2 QLib Alpha158 移植因子集 (新增 ~60个)：**

| 因子类别 | 典型因子 | 说明 |
|---------|---------|------|
| K线形态 | KLEN, KMID, KLOW, KUP | 当日K线形态特征 (上影线、下影线、实体比例) |
| 多尺度动量 | ROC_5/10/20/30/60 | 不同时间窗口的收益率 |
| 多尺度波动 | STD_5/10/20/30/60 | 不同窗口的标准差 |
| 量价关系 | CORR_5/10/20/30 | 价格与成交量相关性 |
| 资金流 | WVMA, VMA_5/10/20 | 加权量能量 |
| 超额收益 | BETA, RSQR, RESI | 相对基准的回归特征 |

> **实现路径**：不直接依赖 QLib 运行时，而是将 Alpha158 的因子表达式 (pandas 向量化运算) 翻译为我们的 `FactorEngine` 接口实现。这样保持架构统一，同时获得 QLib 的因子研究成果。

#### 4.3.1 因子策略引擎

| 属性 | 说明 |
|------|------|
| **职责** | 基于因子组合，对股票池进行打分/排名，生成买卖信号 |
| **方法** | 多因子等权打分 (MVP) → 后续可引入 Qlib/ML 模型 |

**策略流程：**

```
因子矩阵 → 截面标准化 (z-score) → 因子加权打分 → 排名 → 信号状态追踪 → 生成信号
                                                          │
                                        ┌─────────────────┤
                                        ▼                 ▼
                                    买入信号:           卖出信号:
                                    排名前N            排名跌出前M或
                                    (无论是否已      触发止损线
                                     发过信号)
```

> **信号状态追踪**：系统不追踪用户实际持仓（无券商接口，无法可靠获取），而是追踪自身信号历史。每只股票标注信号状态：
> - `new_entry`：首次进入 Top-N（新机会）
> - `sustained`：连续多日在 Top-N 中（维持关注）
> - `exit`：从 Top-N 跌出（卖出信号）
>
> 用户根据信号状态自行判断是新建仓、加仓还是不操作。

**信号输出格式：**

```python
@dataclass
class TradeSignal:
    timestamp: datetime          # 信号时间
    symbol: str                  # 股票代码
    name: str                    # 股票名称
    action: Literal["BUY", "SELL", "HOLD"]  # 操作方向
    signal_status: Literal["new_entry", "sustained", "exit"]  # 信号状态
    days_in_top_n: int           # 连续在 Top-N 中的天数 (0=首次进入)
    price: float                 # 建议价格
    confidence: float            # 置信度 (0-1)
    source: str                  # 信号来源: "factor" | "llm_event" | "merged"
    reason: str                  # 理由描述 (人类可读)
    stop_loss: float | None      # 建议止损价
    take_profit: float | None    # 建议止盈价
    factor_score: float | None   # 因子评分 (用于展示趋势变化)
    prev_factor_score: float | None  # 上期因子评分 (用于展示↑↓)
    metadata: dict               # 扩展信息 (因子值/新闻内容等)
```

#### 4.3.2 LLM 事件驱动引擎

| 属性 | 说明 |
|------|------|
| **职责** | 逐股综合决策：将因子数据+新闻+公告+全球市场打包为"证据包"，LLM 综合判断输出最终 BUY/SELL/HOLD |
| **LLM 路由** | **LiteLLM** — 统一 API，可随时切换 Azure OpenAI/DeepSeek/Gemini/通义等后端 |
| **默认后端** | **Azure OpenAI** (gpt-4o-mini，企业级稳定性 + 合规) |
| **调用模式** | T4 因子选股后逐股调用（每只候选股 1 次 LLM 调用，~20 次/天）。LLM 输出即最终信号，不做硬编码加权融合 |

**多源事件采集架构：**

```
────────────────────────────────────────────────────────────────
 数据源层 (多源并行采集)
────────────────────────────────────────────────────────────────

① 财联社电报 (实时快讯)     → 每小时采集, 全市场快讯
② 东财个股新闻 (个股相关)    → 自选股专属新闻
③ 巨潮公司公告 (定期报告/临时公告) → 100% 上市公司覆盖
④ 国际财经新闻 (全球事件)    → 美联储/关税/财报等全球事件
⑤ Brave Search (网络搜索)      → Phase 2, 补充行业/宏观/国际新闻

────────────────────────────────────────────────────────────────
 决策层 (逐股 LLM 综合决策, 借鉴 PanGus Trader Agent 模式)
────────────────────────────────────────────────────────────────
```

**LLM 逐股综合决策流程（在 T4 中执行，借鉴 PanGus Trader Agent）：**

```
T4 因子选股完成 → 确定因子池 + 自选股 (~20-30 只)
        │
        ▼
① 实时拉取候选股数据
   ├─ 每只股票: 24h 个股新闻 (东财 stock_news_em) → 落库
   ├─ 每只股票: 24h 公告 (巨潮 cninfo) → 落库
   └─ 从 DB 读取: 24h 快讯 (T2 已采集的财联社电报)
        │
        ▼
② 逐股构建"证据包" (~20 只候选股)
   对每只候选股, 构建完整证据 Prompt:
   ├─ 因子数据: 综合得分、截面排名 (Top-N 中第几)、各因子细项
   │   (RSI, MACD, PE, ROE, 宏观因子等)
   ├─ 新闻数据: 该股票 24h 个股新闻 + 公告 + 全量快讯
   └─ 全球市场: 隔夜美股/港股/商品涨跌 (T1 数据)
        │
        ▼
③ 逐股 LLM 综合判断 (~20 次调用)
   LLM 综合所有证据, 模拟三角色:
        │  ┌──────────────────────────────────────────┐
        │  │  🟢 牛方视角: 找利好解读 + 买入理由       │
        │  │  🔴 熊方视角: 找利空解读 + 风险提示       │
        │  │  ⚖️  裁判视角: 综合两方,给出最终判断    │
        │  └──────────────────────────────────────────┘
   注: LLM 自行权衡因子/新闻/全球数据的重要性, 不做硬编码加权
        │
        ▼
④ 结构化输出 (JSON, 每只股票独立)
   ├─ action: BUY / SELL / HOLD (最终建议)
   ├─ confidence: 0.0-1.0 (信心度)
   ├─ bull_reason: 牛方理由
   ├─ bear_reason: 熊方理由
   ├─ judge_conclusion: 裁判结论
   ├─ short_term_outlook: 短期展望 (1-5天)
   └─ mid_term_outlook: 中期展望 (1-3月)
        │
        ▼
⑤ LLM 输出即最终信号 → 推送 (含完整分析理由)
   降级: LLM 全部失败 → 使用纯因子信号 (MultiFactorStrategy 输出)
```

> **为什么让 LLM 做综合决策而非加权融合？** 借鉴 PanGus 的 Trader Agent 设计：把所有证据（因子分数 + 新闻 + 全球市场）交给 LLM，让它自己权衡各信号的重要性。硬编码的 0.6/0.4 权重无法适应不同市场环境（牛市应重因子，黑天鹅时应重事件），而 LLM 的泛化推理能力恰好适合这种"非结构化权衡"。

> **为什么在 T4 而非 T2 中做 LLM？** 因为只有 T4 因子选股完成后才知道需要分析哪些股票，且因子分数是 LLM 证据包的核心输入。如果在 T2 中做，缺少因子数据，LLM 只能做纯事件分析，无法综合判断。

> **事件不参与选股的 trade-off**：因子池纯因子驱动（CSI300 Top-N），事件只在 LLM 综合判断阶段影响最终建议。考虑到：(1) 因子池已是高质量候选，(2) 重大事件往往也会在技术因子端体现（放量、跳空），(3) 自选股始终在分析范围内不会遗漏，这个 trade-off 可以接受。

**Prompt 设计要点：**
- 逐股综合判断：每只候选股一个 prompt，输入为完整"证据包"（因子数据 + 新闻 + 公告 + 快讯 + 全球市场）
- 牛方 (Bull)、熊方 (Bear)、裁判 (Judge) 三角色视角（借鉴 PanGus 多角色辩论）
- 要求输出 JSON 格式：action (BUY/SELL/HOLD) + confidence (0-1) + bull_reason / bear_reason / judge_conclusion
- 因子数据以结构化形式呈现（综合得分、排名、各因子值），LLM 自行判断因子与事件的相对重要性
- 明确要求区分短期 vs 中期展望
- 包含 few-shot examples 提升一致性

**LLM 调用失败链 (Graceful Degradation)：**

```
LLM 调用返回
    │
    ▼
① JSON 校验 (json.loads)
    ├─ 解析成功 → 正常流程
    └─ 解析失败 → ② 重试 1 次
                      ├─ 成功 → 正常流程
                      └─ 仍失败 → ③ LiteLLM fallback 切换下一个 provider
                                    ├─ 成功 → 正常流程
                                    └─ 全部失败 → ④ 规则打分降级
                                                   └─ 关键词正则匹配 (利好词库/利空词库)
                                                      → 简单计数评分 + ⑤ 记录日志 + 推送告警
```

> **降级策略**：当所有 LLM provider 均不可用时，降级为纯因子信号（MultiFactorStrategy 输出的 BUY/SELL）。纯因子模式无牛/熊辩论能力，但保障系统不中断。关键词正则匹配作为最末级 fallback 仅用于新闻有无的判断。

#### 4.3.3 信号输出与安全约束

| 属性 | 说明 |
|------|------|
| **职责** | LLM 输出即最终信号；负责安全约束、去重、降级处理 |
| **决策者** | LLM（综合判断），不做硬编码加权融合 |

**信号生成规则 (MVP)：**

```
LLM 综合决策模式 (正常):
  LLM 接收完整证据包 (因子分+排名+新闻+公告+快讯+全球数据)
  → 输出: action (BUY/SELL/HOLD) + confidence (0-1) + 理由
  → LLM 自行权衡各信号的重要性, 不做硬编码权重

降级模式 (LLM 不可用):
  所有 LLM provider 均失败
  → 降级为纯因子信号 (MultiFactorStrategy 输出)
  → 因子分 > buy_threshold (0.7) → BUY
  → 因子分 < sell_threshold (0.3) → SELL
  → 其他 → HOLD

安全约束:
  - 涨停标的不发买入信号
  - 跌停标的不发卖出信号
  - 单日同一标的同一方向最多推送 1 次 (允许同日先 BUY 后 SELL)

去重与幂等性 (三层保障):
  ① 新闻去重:  source + title_hash (SHA256 前 16 位) → 写入前检查
  ② 信号去重:  symbol + action + date → 同股同方向当日不重复生成
  ③ 推送去重:  symbol + action + date → 同股同方向每日最多推送 1 次
```

### 4.6 任务调度与编排

> 本节定义系统所有定时任务的完整调度方案。任务编排是横切关注点，涉及 Layer 1 到 Layer 5 的所有模块。

#### 4.6.1 调度逻辑

```
调度器每日启动时:
1. 检查 today in trading_days (交易日历, §4.1.6)
2. 交易日 → 执行全部 5 个定时任务 (T1-T5)
3. 非交易日 → 不执行任何任务
```

#### 4.6.2 定时任务总览

APScheduler 注册 5 个任务，按职责解耦为 4 类：

**数据采集 (T1-T3)：**

| # | 任务 ID | 触发时间 | 频率 | 做什么 | 手动触发 |
|---|---------|---------|------|--------|---------|
| **T1** | `sync_global_market` | 08:00 | 每交易日 1 次 | 拉取美股/港股/商品隔夜快照 → 存 SQLite → 计算宏观因子 → 缓存供 T4 使用 | ✅ `run market --global` |
| **T2** | `poll_news` | 07:00–20:00 | 每 **1 小时** | 拉取财联社快讯 → 去重入库 → 清理 >30 天旧新闻（不做 LLM 分析，不拉取个股新闻/公告） | ✅ `run news` |
| **T3** | `sync_domestic_market` | 15:30 | 每交易日 1 次 | 增量同步日 K 线（自选股 + CSI300 全量）+ 基本面数据更新（自选股 + CSI300 全量） | ✅ `run market --domestic` |

> **T2 采集窗口 07:00–20:00 说明**：07:00 开始捕获盘前政策/监管快讯和隔夜国际新闻；延伸到 20:00 覆盖盘后公告密集发布期。T2 只负责快讯的采集和存储，LLM 分析在次日 T4 中执行。

| **T4** | `generate_signals` | 08:15 (T1 完成后) | 每交易日 1 次 | 因子计算 → CSI300+自选股截面排名 → Top-N 因子池 → 实时拉取 24h 新闻/公告（落库）→ 合并 T2 快讯 → 逐股构建证据包（因子+新闻+全球数据）→ **逐股 LLM 综合判断** → 信号事后验证 → 推送"今日操作建议" | ✅ `run signals` |

#### 4.6.3 设计说明

- **T2 只做快讯采集**：T2 只负责拉取财联社快讯并存入 SQLite，不做 LLM 分析。个股新闻和公告的拉取移到 T4（因子选股后按需拉取），避免基于过期 active pool 拉取无关数据。
- **T4 集中处理**：因子选股 → 构建证据包（因子数据+新闻+全球市场）→ 逐股 LLM 综合判断 → 推送。LLM 看到全部证据后自行权衡，输出最终 BUY/SELL/HOLD（借鉴 PanGus Trader Agent 模式）。
- **日线级策略，盘前一天一次**：信号在每交易日 08:15 生成，指导当日交易。不做盘中 5 分钟实时信号（日线级因子和持仓周期不匹配盘中高频更新）。
- **T4 依赖 T1**：信号生成需要全球市场最新数据，T4 在 T1 完成后触发。
- **信号事后验证在 T4 中执行**：每天生成新信号时，同时验证 1/3/5 天前信号的实际收益。
- **日报功能**：不设独立推送任务，T4 产出信号时推送"今日操作建议"（含每只候选股的牛/熊/裁判理由），汇总视图在 Dashboard 看板中展示。

#### 4.6.4 执行时序

**完整交易日时序：**

```
07:00 ─→ T2 快讯采集循环开始 (每小时, 只拉取+存储, 不做 LLM)
              ↓
08:00 ─→ T1 盘前全球市场同步
              ↓
08:15 ─→ T4 信号生成:
              因子选股 → 构建证据包 → 逐股LLM综合判断
              → 事后验证 → 推送"今日操作建议"
              ↓
09:30 ─→ 开盘 (用户根据信号操作)
              ↓
15:00 ─→ 收盘
              ↓
15:30 ─→ T3 盘后国内市场同步 (K线 + 基本面)
              ↓
20:00 ─→ T2 快讯采集循环结束
```

## 7. 数据模型

### 7.1 SQLite 表结构

```sql
-- 日K线数据
CREATE TABLE daily_bars (
    symbol    TEXT NOT NULL,
    date      TEXT NOT NULL,       -- YYYY-MM-DD
    open      REAL,
    high      REAL,
    low       REAL,
    close     REAL,
    volume    INTEGER,
    amount    REAL,                -- 成交额
    adj_factor REAL DEFAULT 1.0,  -- 复权因子
    PRIMARY KEY (symbol, date)
);

-- 新闻/快讯
CREATE TABLE news_items (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp TEXT NOT NULL,
    title     TEXT NOT NULL,
    content   TEXT,
    source    TEXT,
    region    TEXT DEFAULT 'domestic',  -- 地域标记: domestic | global
    symbols   TEXT,                -- JSON array: ["600519", "000858"]
    sentiment REAL,               -- LLM 情感分数 (-1 ~ +1)
    raw_json  TEXT                 -- 原始数据备份
);

-- 交易信号
CREATE TABLE trade_signals (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp  TEXT NOT NULL,
    symbol     TEXT NOT NULL,
    name       TEXT,
    action     TEXT NOT NULL,      -- BUY / SELL / HOLD
    price      REAL,
    confidence REAL,
    source     TEXT,               -- factor / llm_event / merged
    reason     TEXT,
    stop_loss  REAL,
    take_profit REAL,
    pushed     INTEGER DEFAULT 0,  -- 是否已推送
    metadata   TEXT                -- JSON
);

-- 回测结果
CREATE TABLE backtest_results (
    id             INTEGER PRIMARY KEY AUTOINCREMENT,
    run_time       TEXT NOT NULL,
    strategy_name  TEXT,
    config_json    TEXT,
    sharpe_ratio   REAL,
    annual_return  REAL,
    max_drawdown   REAL,
    win_rate       REAL,
    total_trades   INTEGER,
    report_path    TEXT             -- HTML 报告路径
);

-- 基本面数据缓存
CREATE TABLE fundamentals (
    symbol       TEXT NOT NULL,
    date         TEXT NOT NULL,
    pe_ttm       REAL,
    pb           REAL,
    roe_ttm      REAL,
    revenue_yoy  REAL,
    profit_yoy   REAL,
    market_cap   REAL,
    PRIMARY KEY (symbol, date)
);
```

## 9. 风险与待决事项

### 9.1 技术风险

| 风险 | 影响 | 应对 |
|------|------|------|
| AkShare 爬虫接口变更/失效 | 数据中断 | BaoStock 作为回退；数据层 Provider 可切换 |
| 财联社反爬限制 | 新闻源中断 | 降低轮询频率；多源聚合 (东财+巨潮+Brave Search 互备) |
| Azure OpenAI 不稳定/限流 | LLM 分析中断 | LiteLLM 自动 fallback: Azure → DeepSeek → Gemini |
| LLM 幻觉/误判 | 错误信号 | 牛/熊辩论Prompt降低单方面偏见；严格JSON校验；影响度阈值(≥7) |
| 个股新闻稀缺 | 事件引擎无输入 | 3层保障: 公告(100%覆盖)+行业传导+异动价量信号 |
| 回测过拟合 | 实盘亏损 | Phase 2 引入回测 + 样本外测试 + Walk-Forward (PyBroker) |
| 国际行情数据源不稳定 | 宏观因子/异动信号缺失 | AkShare 国际行情接口偏快照式,延迟可接受; 因子引擎受限降级时统一将宏观因子置为中性值 |
| 国际新闻 LLM 推理偏差 | 误判全球事件对 A 股影响 | 牛/熊/裁判 Prompt 多角度降低偏见; 映射表作为上下文辅助 LLM; 逐股分析保证深度 |

### 9.2 业务风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 信号频率过高/过低 | 用户体验差 | 可调阈值 + 每日同股最多1次限制 |
| 因子策略和事件策略冲突频繁 | 产出大量 HOLD | 动态调整权重；增加"强信号"独立触发通道 |
| 免费数据源精度不足 | 因子计算偏差 | Phase 2 切换 Tushare Pro |
